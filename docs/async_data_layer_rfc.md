# RFC: переход data layer на async после стабилизации sync-цепочки

Дата: 2026-04-13  
Статус: Draft / Backlog

## Контекст

В cleanup-итерации мы стабилизировали синхронный запросный pipeline:

`endpoint -> service -> repository -> db session dependency`.

На этом этапе важно не смешивать архитектурную очистку с полной async-миграцией SQLAlchemy,
чтобы не рисковать совместимостью API-контрактов и текущими интеграциями.

## Цель RFC

Отдельно описать путь к async data layer без изменений внешних контрактов API.

## Scope будущей задачи

1. Оценка перехода с Flask-SQLAlchemy scoped session на SQLAlchemy async engine/sessionmaker.
2. Ревизия всех репозиториев на предмет блокирующих операций и I/O.
3. Адаптация сервисного слоя под async интерфейсы репозиториев.
4. Проверка интеграций Celery/фоновых задач на корректность при гибридной sync/async модели.
5. Нагрузочное сравнение latency/throughput до и после.

## Out of scope cleanup-итерации

- Переписывание legacy бизнес-логики на async.
- Массовая смена сигнатур endpoint'ов с sync на async без контрактных тестов.
- Параллельный рефакторинг доменной логики и инфраструктуры DB-драйвера.

## Предварительный план внедрения

1. **PoC на одном read-only сценарии** (`questions_api`) с async репозиторием.
2. **Контрактные тесты + регрессионные smoke** на эквивалентность JSON и кодов ответа.
3. **Фича-флаг/канареечный rollout** для ограниченного процента трафика.
4. **Пошаговый перенос остальных read-path**, затем write-path.
5. **Удаление sync-адаптеров** после стабилизации.

## Риски

- Скрытые блокировки в legacy коде и сторонних SDK.
- Разный lifecycle транзакций между sync и async session.
- Рост сложности при смешанном режиме во время миграции.

## Критерии готовности к старту async-этапа

- [ ] 100% публичных endpoint'ов проходят контрактные тесты в текущем sync-контуре.
- [ ] Репозитории покрывают все SQL-доступы, используемые FastAPI endpoint'ами.
- [ ] Зафиксированы baseline-метрики (latency/error rate) для сравнения.
